# stride 的本质：一块一维内存，如何假装成任意多维张量

> 这篇文章只讲一件事：**stride 到底是什么。**
>
> 为什么改几个整数就能"零拷贝"地转置、切片、广播？又为什么有时候偏偏改不动、非得真的去搬内存？"无脑交换 shape 和 stride 就能实现一切视图变换"——这个直觉错在哪？本文从"物理内存是一维的"这一最朴素的事实出发，把 stride 还原成一道中学级别的多元一次方程，讲透它的设计哲学、数学本质，以及"能做 / 不能做"的那条精确边界。

---

## 一、起点：物理内存永远是一维的

理解一切的第一步，是**抛弃"多维数组"的直觉**。

不管你在 Python 里写的是 `[B, C, H, W]` 还是 `[2, 3]`，落到硬件上，内存永远是一根**线性的、带编号的一维序列**——从下标 0 开始，一个元素挨着一个元素排成一条线。所谓"三维张量""四维张量"，是你在程序里给这条线编的一个故事。

那么，一个多维的逻辑坐标 `(i, j)`，是怎么落到这根线上某个具体位置的？答案就是一套**翻译规则**。一个张量，本质上是这样一个四元组：

```
tensor = ( storage,   # 一段一维内存（真正的数据）
           offset,    # 从第几个元素开始读
           shape,     # 每个维度有多长     [d0, d1, ..., dn-1]
           stride )   # 每个维度走一步，物理下标跨几个元素 [s0, s1, ..., sn-1]
```

后三个（`offset / shape / stride`）都是**元数据**，是"怎么看这段内存"的说明书。`storage` 才是数据本身。

---

## 二、stride 就是那道多元一次方程

把上面的四元组写成公式，逻辑坐标到物理下标的映射长这样：

```
storage_index = offset + i0*s0 + i1*s1 + ... + in-1*sn-1
```

这就是全部。一道再普通不过的**多元一次方程**（线性仿射映射）：

- **`shape`** 是每个坐标 `iₖ` 的**取值范围**（定义域）：第 k 维允许取 0 到 `dₖ-1`。
- **`stride`** 是该维度坐标每 `+1`，物理下标**跨过多少个元素**（这条直线的斜率 / 这个变量的权重）。
- **`offset`** 是整条直线的**截距**（从哪里开始读）。

> **核心洞察**：所谓"视图变换（View）"，在底层编译器和内存管理看来，**根本不触碰物理内存，仅仅是修改了这道方程里的系数（stride）和定义域（shape）**。这就是"零拷贝"的全部秘密——改几个整数，比搬一字节数据便宜几个数量级。一个 `[1024, 1024]` 矩阵转置，真去搬内存是 O(N)；而 `transpose` 只是把 stride 从 `[1024, 1]` 改成 `[1, 1024]`，O(1)。

代价当然有：改完 stride 的张量往往**不再连续**，某些算子会因此变慢或报错。这是后话，第四节展开。

---

## 三、为什么"转置 = 交换 stride"：不过是加法交换律

来看一个具体的、能让你**看见**这道方程的例子。

一段连续内存 `[0, 1, 2, 3, 4, 5]`，我们把它看作一个 `2×3` 矩阵：

```python
>>> a = torch.arange(6).reshape(2, 3)
>>> a
tensor([[0, 1, 2],
        [3, 4, 5]])
>>> a.shape, a.stride(), a.is_contiguous()
((2, 3), (3, 1), True)
```

访问逻辑坐标 `(i, j)` 时，物理偏移是：

```
Offset_old = i*3 + j*1
```

现在对它求**转置**。转置的数学定义是：新矩阵的第 `(x, y)` 个元素，等于老矩阵的第 `(y, x)` 个元素。为了让新坐标 `(x, y)` 命中同一块物理内存里的同一个数据，把 `i = y`、`j = x` 代回原方程：

```
Offset_new = y*3 + x*1 = x*1 + y*3
```

发现了吗？这只是**加法交换律**。对于新的坐标系 `(x, y)`：

- `x`（新行）的系数变成了 `1`；
- `y`（新列）的系数变成了 `3`。

所以新视图的 `stride = (1, 3)`、`shape = (3, 2)`：

```python
>>> t = a.t()
>>> t
tensor([[0, 3],
        [1, 4],
        [2, 5]])
>>> t.shape, t.stride(), t.is_contiguous()
((3, 2), (1, 3), False)
```

这就是为什么"转置在数学上完美等价于交换 shape 和 stride"——底层没有魔法，只是**调换了多元一次方程里两个变量的权重系数**，内存一字节没动。

```
storage = [0, 1, 2, 3, 4, 5]              一维物理内存（永远是线性）

  shape=(2,3), stride=(3,1):              shape=(3,2), stride=(1,3):
    out[i,j] = storage[i*3 + j]             out[x,y] = storage[x*1 + y*3]
    (0,0)->0  (0,1)->1  (0,2)->2            (0,0)->0  (0,1)->3
    (1,0)->3  (1,1)->4  (1,2)->5            (1,0)->1  (1,1)->4
                                            (2,0)->2  (2,1)->5
  → [[0,1,2],[3,4,5]]                     → [[0,3],[1,4],[2,5]]  == a.t()
                  同一段内存，两种读法，零拷贝
```

> **一个必须澄清的陷阱**：`a.view(3, 2)` **不是**转置！`view` 假设内存是行主序，它只改"怎么分组读"，不改"哪个轴是哪个轴"。结果是 `[[0,1],[2,3],[4,5]]`，与 `a.t()` 完全不同：
>
> ```python
> >>> a.view(3, 2)      # 不是转置！只是按行主序重新分组
> tensor([[0, 1],
>         [2, 3],
>         [4, 5]])
> ```
>
> 记住这对最容易混淆的区别：**`view` 改"分组方式"、假设行主序；`transpose/permute` 改"维度顺序"、不假设行主序。** 真正"无脑交换"成立的是后者，不是前者。

## 四、"不能做"的精确含义：仿射映射何时断裂

既然改 stride 这么方便，是不是所有形状变化都能零拷贝完成？**不是。**

这里要纠正一个流传很广的误解：**"被 view 过的张量不能再 view"——这是错的。** 你完全可以无限次叠加纯视图操作，比如 `t.view(2,3).unsqueeze(0).expand(4,2,3).transpose(1,2)`，只要每一步在数学上仍能用"每维一个常数 stride"解释，内存就永远不拷贝。

那真正的边界在哪？一句话：

> **变换后的逻辑坐标 → 物理偏移，必须仍能写成"每个维度一个常数 stride"的那道多元一次方程。** 一旦某个维度需要"变步长"寻址，这道仿射映射就断了，纯视图操作必然失效，只能真的搬内存。

### 失效案例：把转置后的张量展平

接着上面的 `t`（`shape=(3,2)`、`stride=(1,3)`）。逻辑上它的三行是 `[0,3]`、`[1,4]`、`[2,5]`。如果把它**展平成一维**，期望得到：

```
期望逻辑数组 = [0, 3, 1, 4, 2, 5]
```

现在以底层的视角，追踪这 6 个元素在**原始物理内存**里的真实位置（因为 `storage=[0..5]` 且 `offset=0`，值恰好等于物理下标）：

```
期望读 0 -> 物理 0      步长
期望读 3 -> 物理 3       +3
期望读 1 -> 物理 1       -2
期望读 4 -> 物理 4       +3
期望读 2 -> 物理 2       -2
期望读 5 -> 物理 5       +3
```

实跑验证，步长序列正是交替的：

```python
>>> offsets = [0 + i*1 + j*3 for i in range(3) for j in range(2)]   # t 的寻址公式
>>> offsets
[0, 3, 1, 4, 2, 5]
>>> [offsets[k+1] - offsets[k] for k in range(len(offsets)-1)]
[3, -2, 3, -2, 3]
```

步长变成了 **`+3, -2, +3, -2, +3`**，不再是一个常数。

而一维张量在底层数据结构里**只有一个 `stride_0` 整型变量**——它的寻址必须是 `storage_index = offset + k * stride_0` 这条**等差数列**。你要找的物理位置集合 `{0, 3, 1, 4, 2, 5}` 根本不是等差数列，**没有任何一个常数 `stride_0` 能表达这种交替跳跃**。于是仿射映射无解，系统直接报错：

```python
>>> t.view(-1)
RuntimeError: view size is not compatible with input tensor's size and stride
(at least one dimension spans across two contiguous subspaces).
Use .reshape(...) instead.
```

注意 PyTorch 自己给出的提示：**`Use .reshape(...) instead`**——这正是下一节的解法。

### 为什么 `reshape` 不报错？因为它悄悄搬了内存

`reshape` 是 `view` 的"宽容版"。它内部先判能不能 view，能就零拷贝；**不能就自动申请一块新内存，顺着那条 `+3,-2,+3,-2` 的轨迹把数据一个个抠出来、重新写成连续的，再返回**：

```python
>>> r = t.reshape(-1)            # 不报错
>>> r
tensor([0, 3, 1, 4, 2, 5])
>>> r.stride(), r.is_contiguous()
((1,), True)
>>> r.storage().data_ptr() == t.storage().data_ptr()
False                              # False = 发生了真实拷贝，二者不再共享内存
```

对比连续张量上的 `reshape`，那才是真正的零拷贝（共享同一段 storage）：

```python
>>> v = a.reshape(-1)            # a 本来就连续
>>> v
tensor([0, 1, 2, 3, 4, 5])
>>> v.storage().data_ptr() == a.storage().data_ptr()
True                               # True = 零拷贝
```

这就是 `view` / `reshape` / `contiguous()` 三者的本质关系：**`view` 要求连续且零拷贝；`reshape` 不要求连续、可能拷贝；`contiguous()` 必拷贝。**

## 五、边界一句话：2D→2D 总是安全，2D→1D 才会炸

把上面的结论收敛成一句可以秒判的规则：

| 操作 | 维度数变化 | 能否零拷贝 view？ | 原因 |
|---|---|---|---|
| 转置 / permute | 2D→2D | ✅ 能 | 每个维度各自仍是常数 stride，只是把系数换了位置 |
| 转置后展平 | 2D→1D | ❌ 不能 | 把原本 stride 不同的两个维度强行合并进同一维，步长被迫"动态化"，仿射断裂 |

**为什么 2D→2D 总是安全？** 因为转置/permute 只是**重排 stride 数组**，每个维度仍然有一个属于自己的常数 stride，那道多元一次方程依然成立。

**为什么 2D→1D 会炸？** 因为"降到一维"意味着把多个维度**合并（merge）进同一个维度**。合并之所以合法，**充要条件是：被合并的那组维度在物理内存上本来就是挨着的（行主序连续）**——这样它们才能被一条单一的等差数列串起来。而转置/permute 恰恰把这种相邻关系**撕开**了，于是缝合不上。

更一般地，"合并/展平一组维度是合法视图" ⟺ "这组维度在物理内存上本就构成行主序的连续块"。凡是经历过 `transpose / permute / 带 step>1 的 slice` 的维度，其相邻关系已被破坏，**对它们做 flatten / reshape-合并就会失败**。

---

## 六、直觉判据：别去"缝合被撕裂的维度"

写代码时，没人会每次都去脑推多元一次方程。建立一个直观的心智模型就够了。

把张量的数据想象成一根原本整整齐齐的**面条**（物理内存）：

- **🟢 安全（顺水推舟）**：`slice`（截断面条）、`squeeze` / `unsqueeze`（增删长度为 1 的维度，给面条换个包装盒）、不撕裂维度的 `reshape`。面条本身的物理排列顺序没动。
  - 例：`[2, 3, 4]` → `reshape(6, 4)`。`2` 和 `3` 在物理上本就挨着，缝合成 `6` 顺滑无碍，零拷贝。
- **🟡 仍安全（只是换吃法）**：`transpose` / `permute`。本质是改"你吃面条的顺序"，开始跳着吃了。但此时你只是换了记录吃法的"说明书"（shape/stride），面条没动。
- **🔴 引爆点（强行缝合）**：在做过 🟡 危险操作之后，**试图把被撕开的维度展平（flatten）或合并**。
  - 例：`[2, 3]` 转置成 `[3, 2]` 后，非要展平成 `[6]`——相当于要求系统把"跳着吃的轨迹"重新当成一根顺滑面条来对待，它做不到，只能报错或拷贝。

**一句话口诀**：

> **只要你不把"经历过 transpose / permute / step>1 切片"的维度合并（flatten 或 reshape-合并），你的 view 操作就永远安全。**

## 七、工程实践：把判断权交给 `reshape`

理解了底层原理，工程上反而有更省事的办法——**连上面的心智模型都不必每次去想**。

**核心原则：忘掉 `.view()`，前端代码无脑用 `.reshape()`。**

`reshape` 的底层逻辑就是为你处理"能不能 view"这个判断：

1. 能 view → 直接返回视图，零拷贝（等价于调了 `.view()`）；
2. 不能 view（比如刚转置过）→ 自动开新内存、做一次拷贝、整理连续后返回（等价于 `.contiguous().view()`）。

把内存是否连续的判断权交给底层的 C++ 分发，你的代码**永远不会因为 `not contiguous` 而崩溃**。

**如果你在写底层高性能算子、要严格控制内存布局**，那就反过来——把假设写死，依赖 `is_contiguous()`：

```python
assert tensor.is_contiguous(), "Input tensor must be contiguous!"
# 或主动物化
tensor = tensor.contiguous()
```

以下是 `is_contiguous()` 的几种典型取值（已实跑验证）：

```python
>>> a = torch.arange(6).reshape(2, 3)
>>> a.t().is_contiguous()          # 转置 -> False
False
>>> a.permute(1, 0).is_contiguous()# 乱序 -> False
False
>>> a[:, ::2].is_contiguous()      # 带间隔切片 step>1 -> False
False
>>> a.unsqueeze(0).is_contiguous() # 插 1 维 -> True
True
>>> a.view(6).is_contiguous()      # 连续展平 -> True
True
```

## 八、编译器视角：AffineMap 与多面体模型

这套"多元一次方程"并非 PyTorch 的专利。在编译器后端（尤其是 MLIR 这套基础设施里），它被抽象为一个一等公民：**`AffineMap`（仿射映射）**。

编译器把每个张量的寻址、每个循环的下标变换，都表达成 `AffineMap`，进而可以用**多面体模型（Polyhedral Model）**在编译期做循环变换、数据局部性分析、自动并行化——这一切的基础，正是"下标到地址的关系是线性的、可解析的"。

那像 inductor 这类编译器、或是昇腾（Ascend）NPU 这类硬件，**自动化处理非连续张量**时走的是哪条路？两条，且往往并存：

1. **让核"吸收"复杂的 stride**：pointwise / reduction 核在生成时，把每个输入的 stride 直接编进寻址表达式（包括 stride=0 的广播、负 stride 的翻转）。运行期零拷贝，但访存可能不再连续、cache 命中率下降——代价转嫁给了核的访存效率。
2. **编译 pass 自动插入物化指令**：当后端不支持某种 stride（例如部分 NPU path 不支持负 stride 寻址、或某个 matmul 模板强制要求某一维连续）时，编译器会自动插入一个等价于 `contiguous()` 的内存重排核，把非连续张量"摊平"成连续的再喂给后续算子。

所以 stride 的本质决定了它**没有免费的午餐**：你省下的那一次拷贝，要么由核多吃寻址开销来还，要么由 pass 插入的物化核来还。何时吸收、何时物化，是编译器在"访存局部性 vs 拷贝成本"之间的调度权衡。

---

## 九、结语：一句话本质

- **张量的本质**，是一段一维物理内存 + 一张多维仿射寻址地图。
- **能做的事**：只要新的视图能用"每维一个常数 stride"的多元一次方程重新解释同一块内存，就只改地图（shape/stride）、不搬数据，零拷贝。
- **不能做的事**：当新视图要求某个维度以**非固定步长**跳跃访问内存时（最典型的就是"把转置/permute 后被撕开的维度展平合并"），仿射地图画不出来，`stride` 机制兜不住，必须做物理重排（拷贝）。

所以，"无脑交换 shape 和 stride 就能实现一切视图变换"是个错觉。stride 能优雅表达的，只有"每个维度仍是常数步长"的那部分世界；一旦某个维度的步长需要随下标变化，这道线性方程就无解了——而那一刻，正是你该把 `.view()` 换成 `.reshape()`（或显式 `.contiguous()`）、让系统去搬内存的信号。
