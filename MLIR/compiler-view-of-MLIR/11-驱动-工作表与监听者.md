# 第 11 章 驱动、工作表与监听者

> **本章位置**　第 9 章讲了 matchAndRewrite 的全生命周期，第 10 章讲了重写的原子动作（RAUW + erase）。本章讲重写的"调度层"——驱动（driver）、工作表（worklist）与监听者（listener）。这里有一个 Dragon Book 不深究、但 MLIR 必须面对的理论——**不动点与单调框架**。本章补这个理论，并讲 PatternBenefit、fold 先于 pattern、DRR/PDL 声明式写法。
>
> **前置依赖**　第 9 章（matchAndRewrite、driver 主循环）、第 10 章（RAUW/erase，listener 在它们里发通知）。
>
> **编译原理切入**　本章从**重写系统的终止性与不动点**立论。greedy driver 的收敛本质是"反复应用规则到 IR 不再改变"——这正是单调框架下数据流分析迭代到不动点的对偶。Dragon Book 第 9 章用一整章讲数据流分析的不动点理论（迭代到收敛、单调框架、Tarski 不动点定理），但它的对象是"信息"（每个点的到达定值集合），MLIR 的对象是"IR 形态"。两者都是单调框架下的迭代到不动点。本章补这个理论背景，让读者理解"为什么 greedy driver 能终止"。

## 11.1 重写的收敛：不动点的直觉

第 9 章讲过 greedy driver 的主循环：pop op → 试规则 → 改图 → 把受影响 op 重塞工作表 → 继续。这个循环什么时候停？

答案是：**当一轮完整遍历后，IR 没有任何变化**——这就是收敛（fixpoint）。

[GreedyPatternRewriteDriver.cpp:826-894](../MLIR-Tutorial/third_party/llvm-project/mlir/lib/Transforms/Utils/GreedyPatternRewriteDriver.cpp#L826) 的外层 `do-while`：

```cpp
LogicalResult simplify(...) && {
  bool continueRewrites = false;
  do {
    if (++iteration > config.maxIterations) break;   // 迭代上限（默认 10，防不收敛）
    worklist.clear();
    region.walk([&](Operation *op){ addToWorklist(op); });  // 每轮重新入表
    continueRewrites = processWorklist();             // 跑这一轮
    continueRewrites |= succeeded(simplifyRegions(rewriter, region, ...));
  } while (continueRewrites);                         // 只要这轮有变化，就再来一轮
  return success(!continueRewrites);                  // 上一轮无变化 = 收敛
}
```

**收敛（fixpoint）= "跑一轮，IR 一点没变"**。这正是 progressive lowering 的运转方式——规则反复套用，直到这组规则再也改不动为止。

> **编译原理浸润点：不动点与单调框架**　Dragon Book 第 9 章的数据流分析也是"迭代到不动点"——反复应用传递函数，直到每个点的信息集合不再变化。两者共享同一套数学基础：
>
> - **Tarski 不动点定理**：在完全偏序集上，单调函数有最小不动点和最大不动点。数据流分析求的是最大或最小不动点（取决于分析方向）。
> - **单调框架**：传递函数必须单调（只增加或只减少信息，不反复横跳），保证迭代必然收敛。
>
> MLIR 的重写不是求数值不动点，而是求"IR 形态不动点"——反复应用重写规则直到 IR 不再改变。它的"单调性"来自一个事实：**每条规则的应用都严格减少了某个度量**（如 op 总数、或某个复杂度评分），所以总重写次数有上限。这就是 §11.5 讲的"终止性"的数学基础。Dragon Book 第 9 章讲的"数据流分析迭代到不动点"与 MLIR 的"重写到不动点"是同一思想的两个应用——一个是信息空间，一个是 IR 空间。

## 11.2 监听者机制的回顾

第 9 章已讲了"驱动就是 rewriter 的监听者"——这是重写调度的核心设计。这里回顾要点，为 §11.3 的设计契约铺垫。

```cpp
// GreedyPatternRewriteDriver : public RewriterBase::Listener
GreedyPatternRewriteDriver(...) : rewriter(ctx), matcher(patterns) {
  matcher.applyDefaultCostModel();   // 按 benefit 排序
  rewriter.setListener(this);        // ★ 驱动把自己设成 rewriter 的监听者
}
```

pattern 里每调一次 `rewriter.xxx()`，rewriter 回调驱动的 `notifyXxx`：

| pattern 调的 | rewriter 发的通知 | 驱动听见后做 |
|---|---|---|
| `rewriter.create<>()` | notifyOperationInserted | addToWorklist(newOp) |
| `rewriter.replaceOp()` | notifyOperationReplaced（转发给 config.listener）+ 间接经 eraseOp 触发 notifyOperationErased | addOperandsToWorklist + worklist.remove |
| `rewriter.eraseOp()` | notifyOperationErased | addOperandsToWorklist + worklist.remove |
| `rewriter.modifyOpInPlace()` | notifyOperationModified | addToWorklist |
| `rewriter.notifyMatchFailure()` | （不改图，只通知） | 记 debug 日志 |

> ⚠️ **精度修正**：`notifyOperationReplaced`（[GreedyPatternRewriteDriver.cpp:754-762](../MLIR-Tutorial/third_party/llvm-project/mlir/lib/Transforms/Utils/GreedyPatternRewriteDriver.cpp#L754)）**本身不动工作表**，它只是把通知转发给可选的 `config.listener`。replace 的工作表更新是**间接**发生的——因为默认 `replaceOp` 内部最终调了 `eraseOp`（第 10 章 §10.4），`eraseOp` 触发的 `notifyOperationErased` 才真正改工作表。别误以为"replace 直接刷新工作表"。

## 11.3 设计契约（matchAndRewrite 的游戏规则）

用 matchAndRewrite 有几个必须遵守的契约。第 9 章已点过，这里展开。

### 11.3.1 契约①：match-before-mutate（先确认能换，再动手）

**这是 matchAndRewrite 最重要的一条规矩**：在 `matchAndRewrite` 里，**一旦你调了任何会改图的 rewriter 方法（create/replace/erase/modifyOpInPlace），就视为你已"承诺"本次一定成功**。不要"改了一半发现不行又 `return failure()`"。

为什么？看下一条。

### 11.3.2 契约②：greedy 驱动没有回滚

`PatternRewriter` 唯一新增的钩子 `canRecoverFromRewriteFailure()` **默认返回 `false`**（[PatternMatch.h:794](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/IR/PatternMatch.h#L794)）。含义是：**如果 pattern 改了图又返回失败，驱动无法撤销**——因为 greedy 用的 `replaceOp/eraseOp` 是立即生效的（第 10 章），没留撤销日志。

对比 dialect conversion 框架的 `ConversionPatternRewriter`：

| | greedy `PatternRewriter`（本章） | `ConversionPatternRewriter`（lowering） |
|---|---|---|
| `canRecoverFromRewriteFailure()` | **`false`** | **`true`** |
| 改图方式 | 立即生效、不记账 | 记"动作日志"，可整体回滚（undo） |
| pattern 改一半失败 | IR 处于半改状态（bug） | 自动回滚到改之前 |
| 适用 | 局部化简、peephole、同方言优化 | 跨方言 lowering（需要"全做或全不做"的原子性） |

> **所以 §11.3.1 的规矩可以这样理解**：greedy 驱动把"无回滚"作为既定约束，把责任交给 pattern——**你要么先把条件查清楚再动手（推荐两步 `match`/`rewrite` 写法，结构性保证不会"改了又失败"），要么在 `matchAndRewrite` 里把所有可能失败的前置检查放在任何 rewriter 调用之前**。这是把"一个看似的限制"变成了"一条清晰的纪律"。

#### 失败的正确姿势：notifyMatchFailure

放弃匹配时，**不要写裸 `return failure()`**，要写：

```cpp
return rewriter.notifyMatchFailure(op, "operands not all from the same defining op");
// 或带详细诊断的回调形式
return rewriter.notifyMatchFailure(op, [&](Diagnostic &d){ d << "..."; });
```

`notifyMatchFailure`（[PatternMatch.h:716-740](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/IR/PatternMatch.h#L716)）做两件事：① 把失败原因转发给 listener（驱动据此在 `-debug` 下打印"为什么这条规则没套上"）；② **返回 `failure()` 给你**。所以它等价于 `return failure()` 但额外送你一条可观测的诊断。

> **这是从第一天就该养成的调试习惯**：当你的 pattern 在 lit 测试里"该匹配却没匹配"时，`-debug-only=greedy-rewriter` 配合 `notifyMatchFailure` 能直接告诉你每条规则在每个 op 上失败的原因。

### 11.3.3 契约③：benefit 是静态的、决定尝试顺序

- **范围**：0..65534，65535 是 `impossibleToMatch` 哨兵（[PatternMatch.h:35](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/IR/PatternMatch.h#L35)）。**越大越先试**。
- **静态**：一个 pattern 实例的 benefit 在构造时定死，**不会随 IR 变化**。源码注释说：若一条规则的"收益"取决于运行时 IR，你应该把它拆成多个 pattern 实例（不同 benefit）、用不同的前置谓词守护。
- **平手按注册顺序**：benefit 相同时，`stable_sort` 保持注册顺序——别依赖语义无关的顺序。

> **benefit 与合流性的关系**：第 9 章讲过项重写系统的合流性（Church-Rosser）——规则应用顺序不影响最终结果。这意味着 **benefit 只影响效率（走几步到达不动点），不影响正确性（最终 IR 一样）**。所以 benefit 的调优是性能工程，不是正确性问题。这是把"一个工程旋钮"建立在"一个数学保证"之上的优雅设计。

### 11.3.4 契约④：改图必须走 rewriter（别绕过）

在 pattern 里**永远不要**直接调 `op->erase()`、`op->setAttr(...)`、`op->setOperand(...)`——它们不发通知，驱动的工作表会失同步（指向已删 op、或漏掉该重访的 op）。要用：

| 想做的事 | ✅ 走 rewriter | ❌ 别直接调 |
|---|---|---|
| 删 op | `rewriter.eraseOp(op)` | `op->erase()` |
| 改 op 属性/操作数（原地） | `rewriter.modifyOpInPlace(op, [&]{ op->setAttr(...); })` | `op->setAttr(...)` |
| 建 op | `rewriter.create<>(...)` | `OpBuilder`（除非你手动挂 listener） |

`modifyOpInPlace`（[PatternMatch.h:629](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/IR/PatternMatch.h#L629)）是 `startOpModification` + 你的 lambda + `finalizeOpModification` 的 RAII 包装，`finalizeOpModification` 会发 `notifyOperationModified`——所以原地改的 op 会正确地被重塞工作表。

## 11.4 终止性：两条"不会无限循环"的保险

pattern 可能"换出来的 op 又匹配自己"（如 `reshape(reshape(x))→reshape(x)` 换完还是 reshape）。MLIR 用两条保险防止无限循环：

1. **driver 层**：`config.maxIterations`（默认 10，[GreedyRewriteConfig:66](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/Transforms/GreedyPatternRewriteDriver.h#L66)）限制外层 do-while 轮数；`config.maxNumRewrites`（默认无限制）限制总重写次数。NorthStar `DeviceRegionFusion.cpp:268` 显式设了 `maxIterations = 10`。
2. **pattern 层**：若你的 pattern 确实会递归套用自己的输出、但**已知有界**，可在 pattern 里调 `setHasBoundedRewriteRecursion()`（[PatternMatch.h:202](../MLIR-Tutorial/third_party/llvm-project/mlir/include/mlir/IR/PatternMatch.h#L202)）告诉驱动"放心，我会收敛"。否则驱动对某些自递归会保守处理。

> **编译原理浸润点：项重写系统的终止性**　Baader & Nipkow 的 *Term Rewriting and All That* [Baader 1998] 用大量篇幅讨论"如何证明一个重写系统终止"——常见的有**字典序终止法**（定义项的复杂度，证明每次重写严格减小复杂度）、**多项式解释法**（把项解释成多项式，重写使多项式值严格减小）。MLIR 的 driver 用的是工程手段（maxIterations + bounded 申明），不是数学证明——它假设规则集是终止的（实际几乎总是），但用 maxIterations 兜底防止意外。这是"工程保险"与"数学保证"的分工：规则作者负责设计终止的规则（或申明 bounded），driver 负责用 maxIterations 兜底。

## 11.5 fold 先于 pattern：更便宜的"局部化简"

第 9 章 §9.5 讲过主循环四步——pop → 死代码 → **fold** → 规则。这里展开 fold。

每个 op 在套用 rewrite pattern 之前，会先被 `fold` 试一遍（[GreedyPatternRewriteDriver.cpp:491-563](../MLIR-Tutorial/third_party/llvm-project/mlir/lib/Transforms/Utils/GreedyPatternRewriteDriver.cpp#L491)）。fold 和 pattern 是两种不同的化简机制：

| | `fold()` | rewrite pattern |
|---|---|---|
| 写在哪 | Op 的成员（ODS 生成骨架，如 `MyOp::fold(FoldAdaptor)`） | 独立的 `OpRewritePattern` |
| 输入/输出 | 看 op 自身 + 操作数的"已知值"，返回 `OpFoldResult`（Attribute 或 Value） | 任意逻辑，用 rewriter 改图 |
| 改图能力 | 弱：只能"把结果换成常量/已知值"或"原地小改" | 强：能建任意新 op、重组结构 |
| 在循环里的位置 | **先**（[GreedyPatternRewriteDriver.cpp:493](../MLIR-Tutorial/third_party/llvm-project/mlir/lib/Transforms/Utils/GreedyPatternRewriteDriver.cpp#L493)） | 后（[GreedyPatternRewriteDriver.cpp:614](../MLIR-Tutorial/third_party/llvm-project/mlir/lib/Transforms/Utils/GreedyPatternRewriteDriver.cpp#L614)） |
| 专属陷阱 | `ConstantLike` 的 op 不 fold（否则常量折叠成 Attribute 又被物化成常量 op，无限循环） | 自递归要声明 bounded |

> **为什么 fold 先于 pattern？**　因为 fold 更便宜——它不需要建新 op、不需要触发 listener 通知、不需要更新工作表。如果 fold 能搞定（如 `addi(1, 2)` → `3`），就不必动用更重的 pattern。这是"先试便宜的、再试贵的"的工程优化。Dragon Book 第 8.5 节的常量折叠就是 fold 的经典例子。MLIR 把它和 pattern 放在同一驱动里协作：fold 搞不定的，才轮到 rewrite pattern。

## 11.6 同一条规则的三种写法：C++ / DRR / PDL

同一条 `reshape(reshape(x)) → reshape(x)` 规则，MLIR 有三种表达力递增的写法：

**① 手写 C++（本书主线）**：`OpRewritePattern` + `matchAndRewrite`（如 `SimplifyRedundantTranspose`）。最灵活，能写任意控制流。

**② DRR（Declarative Rewrite Rules，TableGen）**——声明式写法：

```tablegen
// ToyCombine.td:33-34
def ReshapeReshapeFoldPattern : Pat<
    (Tensor_ReshapeOp (Tensor_ReshapeOp $arg0, $shape0), $shape1),
    (Tensor_ReshapeOp $arg0, $shape1)>;
```

DRR 用 TableGen 语法描述"左半边模式 + 右半边替换"，`mlir-tblgen` 生成等价的 C++ `OpRewritePattern`。好处是声明式——不必写控制流，编译器生成代码。代价是不如 C++ 灵活（复杂模式写不了）。

**③ PDL（Pattern Descriptor Language）**——更高级的声明式，用 `.mlir` 语法描述模式，运行时解释执行。PDL 还能配合 PDL_interp 写复杂约束。

三者关系：DRR 生成 C++ pattern；PDL 是运行时解释的更抽象层。本书主线是手写 C++（最能看清机制），DRR/PDL 是工程上的便利手段。

> **编译原理浸润点：声明式 vs 命令式重写**　这个三种写法的递进，对应编译器领域的"声明式 vs 命令式"经典权衡。手写 C++ 是命令式（你写控制流），DRR/PDL 是声明式（你描述模式，工具生成/解释）。声明式的好处是简洁、可优化（工具能做模式分析）、可校验；代价是表达力受限。Dragon Book 第 8 章的语法制导翻译是声明式的早期例子。MLIR 同时支持三种，让用户按需选择——简单模式用 DRR，复杂模式用手写 C++，跨方言批量用 PDL。

## 11.7 Pass 集成：从 pattern 到驱动

pattern 写完，要在 Pass 里"装弹 + 开火"。四步范式（所有基于 greedy driver 的 Pass 都长这样）：

```cpp
void DeviceRegionFusionPass::runOnOperation() {
  // ① 建 pattern 集合
  RewritePatternSet patterns(&getContext());
  populateDeviceRegionFusionPatterns(patterns);
  // ② 冻结（不可变快照，驱动拿的是它）
  // ③ 配置（maxIterations / useTopDownTraversal / strictMode / scope）
  GreedyRewriteConfig cfg; cfg.maxIterations = 10;
  // ④ 开火（进入第 9 章的驱动主循环）
  applyPatternsAndFoldGreedily(getOperation(),
      FrozenRewritePatternSet(std::move(patterns)), cfg);
}
```

| 步 | 做什么 | 涉及的类 |
|---|---|---|
| ① 建 pattern 集合 | `RewritePatternSet patterns(ctx);` 然后 `populateXxxPatterns(patterns);` | `RewritePatternSet` |
| ② 冻结 | `FrozenRewritePatternSet frozen(std::move(patterns));` | `FrozenRewritePatternSet`（不可变快照） |
| ③ 配置 | `GreedyRewriteConfig cfg;` | `GreedyRewriteConfig` |
| ④ 开火 | `applyPatternsAndFoldGreedily(op, frozen, cfg);` | 进入驱动主循环 |

`populateXxxPatterns` 是约定俗成的"把一族 pattern 注册进集合"的辅助函数，内部就是 `patterns.add<XxxPattern>(ctx, benefit);`。

## 11.8 与 lowering 的衔接

本书第六部分（Ch9-11）讲的是**同方言重写**——规则在一个 dialect 内改 op。MLIR 还有**跨方言 lowering**（Conversion 框架），它把高层 dialect 的 op 换成低层 dialect 的 op（如 `arith.addi` → `llvm.add`）。两者的关键区别：

| | greedy 重写（本章） | Conversion（lowering） |
|---|---|---|
| rewriter | `PatternRewriter`（无回滚） | `ConversionPatternRewriter`（有 undo 日志） |
| 改图失败 | 半改状态（bug） | 自动回滚 |
| 适用 | 同方言局部化简 | 跨方言"全做或全不做" |

lowering 的完整实战（CH-12 of NorthStar）超出本书范围，但它的核心动词仍是重写——只是用了一个更复杂的 rewriter（带 undo）。理解了本章的 greedy driver，到 lowering 时只需理解"多了一层 undo 日志"。

## 编译原理浸润点回顾

1. **不动点与单调框架**：本章主题。greedy driver 收敛 = IR 形态不动点。与 Dragon Book 第 9 章数据流不动点对偶。Tarski 不动点定理是数学基础。
2. **项重写终止性**：Baader & Nipkow 的终止性证明方法（字典序、多项式解释）。MLIR 用 maxIterations + bounded 申明做工程保险。
3. **合流性（回扣 Ch9）**：benefit 只影响效率不影响正确性，因 Church-Rosser。
4. **常量折叠（fold）**：Dragon Book 第 8.5 节。MLIR 把它和 pattern 放在同一驱动协作。
5. **声明式 vs 命令式重写**：DRR/PDL（声明式）vs 手写 C++（命令式）。Dragon Book 第 8 章语法制导翻译的传统。
6. **数据流分析的迭代算法（回扣 Ch5）**：worklist driver 是它在"重写"上的对偶。

---

## 本章关键结论

1. **重写收敛 = IR 形态不动点**。外层 do-while 直到"一轮无变化"。这与 Dragon Book 第 9 章数据流不动点对偶，共享 Tarski 不动点定理的数学基础。
2. **驱动作为监听者**自动维护工作表。pattern 改图必须走 rewriter，否则通知不发、工作表失同步。
3. **四条设计契约**：match-before-mutate、无回滚、benefit 静态决定顺序、改图必须走 rewriter。失败用 notifyMatchFailure 不用裸 failure()。
4. **终止性靠 maxIterations + bounded 申明**。数学上是项重写系统的终止性（Baader & Nipkow）。
5. **fold 先于 pattern**：更便宜的局部化简（常量折叠）。两者在同一驱动协作。
6. **三种写法**：手写 C++（命令式，最灵活）/ DRR（TableGen 声明式）/ PDL（运行时解释）。benefit 只影响效率不影响正确性（合流性）。
7. **Pass 集成四步**：建集合 → 冻结 → 配置 → 开火。
